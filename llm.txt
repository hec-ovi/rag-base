# rag-base - Technical Reference

## Purpose

Standalone containerized RAG backend. Hybrid retrieval (semantic + keyword + graph) with embedding and reranking, exposed as a REST API.

## Why this exists

Every RAG project needs the same infrastructure: vector storage, embeddings, reranking, search. This solves that once as a plug-and-play service. Future projects (legal, medical, etc.) spin this up and only write the frontend/business logic.

---

## How it works

### Ingest pipeline

1. Client sends document to `POST /v1/documents`
2. API chunks the text (structure-aware, paragraph boundaries, configurable size/overlap)
3. **Contextual Chunk Headers (CCH)**: document title + metadata are prepended to each chunk before embedding (e.g. `[React Documentation | source: docs] chunk text...`). This gives the embedding model document-level context per chunk (~28% retrieval improvement). The stored chunk keeps the original text for display.
4. API embeds all enriched chunks in one batch via TEI (`POST /embed`)
5. API stores document + chunks + vectors + tsvector in PostgreSQL in a **single atomic transaction** - if embedding fails, nothing is committed
6. Graph entities/relations are created separately via `/v1/concepts` and `/v1/relations`

### Retrieval pipeline

1. Client sends query to `POST /v1/search`
2. API embeds the query via TEI
3. Three retrievers run **in parallel**:
   - **Semantic**: pgvector cosine similarity (`embedding <=> query_vector`) - finds content that means similar things
   - **Keyword**: PostgreSQL tsvector full-text search (`tsv @@ websearch_to_tsquery`) - finds exact term matches
   - **Graph**: Memgraph Cypher traversal - expands entities found in the query by 1-2 hops to find related concepts
4. Results merged via **RRF** (Reciprocal Rank Fusion): `score(d) = sum(1/(60 + rank_i))` per retriever - results found by multiple retrievers get boosted
5. Top N candidates sent to TEI reranker (`POST /rerank`) - a cross-encoder that scores query-document pairs much more accurately than cosine similarity
6. Final top K returned to client

### Why this order

- Parallel fan-out: each retriever catches what others miss (proven 5-15% better than single retriever)
- RRF before reranking: the reranker needs to see ALL candidates to make the best decision
- Rerank last: cross-encoder is the most accurate scoring but slowest, so only run it on the merged top-N, not on everything

## Containers

### postgres (pgvector/pgvector:0.8.2-pg17)
- **Port**: 5433 host → 5432 internal (configurable via `POSTGRES_PORT`)
- **Extensions**: vector (pgvector 0.8.2), pg_trgm (fuzzy matching)
- **Schema**: `documents` table, `chunks` table (with `embedding vector(1024)` and `tsv tsvector` columns)
- **Indexes**: HNSW on `chunks.embedding` (cosine distance), GIN on `chunks.tsv` (full-text), GIN on `documents.metadata` (JSONB)
- **Volume**: `ragbase_pgdata` (named Docker volume)
- **Init**: `postgres/init.sql` runs on first boot only via `/docker-entrypoint-initdb.d/`

### embedding (ghcr.io/huggingface/text-embeddings-inference:cpu-1.9)
- **Port**: 8081 host → 80 internal (configurable via `EMBEDDING_PORT`)
- **Default model**: `Snowflake/snowflake-arctic-embed-l-v2.0` (568M params, 1024 dims)
- **Model cache**: host dir `EMBEDDING_MODELS_DIR` mounted at `/data` inside container
- **First boot**: downloads model from HuggingFace Hub (~1.2 GB). Takes 1-3 min. Cached after.
- **TEI endpoints**: `POST /embed`, `POST /v1/embeddings` (OpenAI-compatible), `GET /health`, `GET /info`

### reranker (ghcr.io/huggingface/text-embeddings-inference:cpu-1.9)
- **Port**: 8082 host → 80 internal (configurable via `RERANK_PORT`)
- **Default model**: `BAAI/bge-reranker-v2-m3` (568M params, multilingual)
- **Docker profile**: `rerank` - only starts with `docker compose --profile rerank up -d`
- **Optional**: API works without it, skips reranking step
- **TEI endpoints**:
  - `POST /rerank` - request: `{"query": "...", "texts": ["...", "..."]}` → response: `[{"index": 0, "score": 0.95}, ...]`
  - `GET /health`, `GET /info`

### memgraph (memgraph/memgraph-mage:3.9.0)
- **Port**: 7687 host → 7687 internal (configurable via `MEMGRAPH_PORT`)
- **Protocol**: Bolt only (no REST). Connected via `neo4j` Python async driver with `bolt://memgraph:7687`
- **Docker profile**: `graph` - only starts with `docker compose --profile graph up -d`
- **Optional**: API works without it, graph endpoints return 503
- **Algorithms** (via MAGE):
  - `CALL pagerank.get() YIELD node, rank`
  - `CALL community_detection.get() YIELD node, community_id`
  - BFS shortest path: `MATCH p = (a)-[*BFS]-(b) RETURN p`
- **Volume**: `ragbase_mgdata` (named Docker volume)
- **Constraints**: unique constraint on `Concept.name` (created at API startup)

### api (custom, python:3.12-slim-bookworm)
- **Port**: 5050 host → 8000 internal (configurable via `API_PORT`)
- **Stack**: FastAPI, asyncpg (Postgres), httpx (TEI), neo4j async driver (Memgraph), uvicorn + uvloop
- **Depends on**: postgres (healthy) and embedding (healthy) - waits for both before starting
- **Optional services**: tries to reach reranker and memgraph at startup. If unreachable, logs a warning and disables them. **Does not crash.**
- **Healthcheck**: `GET /health` on internal port 8000

---

## Startup order

1. **Postgres** starts. Runs `init.sql` on first boot (creates tables, indexes). Healthcheck: `pg_isready`.
2. **TEI embed** starts in parallel. Downloads model on first boot (1-3 min). Healthcheck: `GET /health` on port 80.
3. **TEI rerank** (if `--profile rerank`) starts in parallel. Downloads its own model.
4. **Memgraph** (if `--profile graph`) starts in parallel.
5. **API** starts **last** - Docker `depends_on` waits for Postgres and TEI embed to be healthy.
6. On startup, API probes reranker and memgraph. If unreachable → logs warning, sets them to disabled. If reachable → connects.

**If the reranker finishes downloading after the API starts**: `docker compose restart api`

## API Reference

Base URL: `http://localhost:5050`
OpenAPI docs: `http://localhost:5050/docs`

### POST /v1/documents

Ingest a document. Chunks it, embeds each chunk, stores atomically.

**Request:**
```json
{
  "title": "React Documentation",
  "content": "React is a JavaScript library for building user interfaces...",
  "metadata": {"source": "docs", "language": "en"}
}
```

**Response (201):**
```json
{
  "id": 1,
  "title": "React Documentation",
  "content": "React is a JavaScript library for building user interfaces...",
  "metadata": {"source": "docs", "language": "en"},
  "created_at": "2026-04-11T10:29:22.526748Z",
  "updated_at": "2026-04-11T10:29:22.526748Z",
  "chunk_count": 1
}
```

**Error responses:**
- `422` - empty title or empty content

### GET /v1/documents?offset=0&limit=20

List documents, paginated. Returns array of document objects with `chunk_count`.

### GET /v1/documents/{id}

Get a document with all its chunks.

**Response (200):**
```json
{
  "id": 1,
  "title": "React Documentation",
  "content": "...",
  "metadata": {...},
  "created_at": "...",
  "updated_at": "...",
  "chunk_count": 1,
  "chunks": [
    {"id": 1, "chunk_index": 0, "content": "React is a JavaScript...", "token_count": 85}
  ]
}
```

**Error responses:**
- `404` - document not found

### DELETE /v1/documents/{id}

Delete document and all its chunks (CASCADE). Returns `204 No Content`.

**Error responses:**
- `404` - document not found

### POST /v1/search

Hybrid search: semantic + keyword + graph expansion + reranking.

**Request:**
```json
{
  "query": "frontend frameworks for building UI",
  "top_k": 10,
  "rerank": true,
  "rerank_candidates": 50,
  "min_score": 0.0,
  "include_graph": true
}
```

| Field | Type | Default | Description |
|---|---|---|---|
| `query` | string | *(required)* | Search query text |
| `top_k` | int | 20 | Number of results to return (1-100) |
| `rerank` | bool | true | Use cross-encoder reranking (falls back if reranker disabled) |
| `rerank_candidates` | int | 50 | How many RRF-merged results to send to reranker |
| `min_score` | float | 0.0 | Minimum cosine similarity for semantic results |
| `include_graph` | bool | true | Include graph expansion (ignored if Memgraph disabled) |

**Response (200):**
```json
{
  "query": "frontend frameworks for building UI",
  "results": [
    {
      "chunk_id": 4,
      "document_id": 5,
      "document_title": "React Documentation",
      "content": "React is a JavaScript library for building user interfaces...",
      "score": 0.9924,
      "sources": ["semantic", "keyword"]
    }
  ],
  "total": 3,
  "retrievers_used": ["semantic", "keyword", "graph", "rerank"]
}
```

`sources` shows which retrievers contributed to each result. `retrievers_used` shows all retrievers that ran.

### POST /v1/search/semantic

Vector-only search. Same request/response shape. No keyword, graph, or reranking.

### POST /v1/search/keyword

Keyword-only search. Same request/response shape. No vector, graph, or reranking.

### POST /v1/concepts

Create or update a concept node in the knowledge graph. **Upserts by name** - if a concept with the same name exists, it updates type/description/metadata instead of creating a duplicate.

**Request:**
```json
{"name": "React", "type": "Framework", "description": "JavaScript UI library", "metadata": {}}
```

**Response (201):**
```json
{"id": 9, "name": "React", "type": "Framework", "description": "JavaScript UI library", "metadata": {}}
```

### GET /v1/concepts/{id}

Get a concept with all its outgoing relations.

**Response (200):**
```json
{
  "id": 9,
  "name": "React",
  "type": "Framework",
  "description": "JavaScript UI library",
  "metadata": {},
  "relations": [
    {"relation_type": "DEPENDS_ON", "target": "JavaScript", "target_id": 10}
  ]
}
```

**Error responses:**
- `404` - concept not found
- `503` - Memgraph is disabled

### DELETE /v1/concepts/{id}

Delete concept and all its edges (DETACH DELETE). Returns `204`.

### POST /v1/relations

Create a typed directed edge between two concepts (by name).

**Request:**
```json
{"source_name": "React", "target_name": "JavaScript", "relation_type": "DEPENDS_ON", "metadata": {}}
```

**Error responses:**
- `400` - self-referencing relation (source_name == target_name)
- `404` - one or both concepts not found
- `503` - Memgraph is disabled

### GET /v1/relations?concept_name=React

Get all relations involving a concept (both directions).

### DELETE /v1/relations/{id}

Delete a relation. Returns `204`.

### GET /v1/graph/neighbors/{concept_id}?depth=2

Multi-hop neighborhood traversal. Returns all concepts within N hops (max 5).

**Response (200):**
```json
[
  {"id": 10, "name": "JavaScript", "type": "Language", "depth": 1},
  {"id": 12, "name": "Node.js", "type": "Runtime", "depth": 2}
]
```

### GET /v1/graph/path/{from_id}/{to_id}

Shortest path between two concepts (BFS).

**Response (200):**
```json
[
  {"id": 11, "name": "Next.js", "type": "Framework"},
  {"id": 9, "name": "React", "type": "Framework"},
  {"id": 10, "name": "JavaScript", "type": "Language"}
]
```

**Error responses:**
- `404` - no path exists between the two concepts

### GET /v1/graph/communities

Louvain community detection. Returns every concept with its community ID.

**Response (200):**
```json
[
  {"id": 13, "name": "PostgreSQL", "type": "Database", "community_id": -1},
  {"id": 9, "name": "React", "type": "Framework", "community_id": 0},
  {"id": 10, "name": "JavaScript", "type": "Language", "community_id": 0}
]
```

Isolated nodes get `community_id: -1`.

### GET /v1/graph/stats

```json
{"concepts": 6, "relations": 6}
```

### POST /v1/embed

Embed text(s) directly via TEI. Passthrough.

**Request:**
```json
{"inputs": ["React is a JavaScript library", "PostgreSQL is a database"]}
```

**Response (200):**
```json
{
  "embeddings": [[0.23, -0.41, ...], [0.11, 0.55, ...]],
  "model": "Snowflake/snowflake-arctic-embed-l-v2.0",
  "dimensions": 1024
}
```

### POST /v1/rerank

Rerank candidates directly via TEI. Passthrough.

**Request:**
```json
{"query": "frontend frameworks", "texts": ["React is...", "PostgreSQL is..."], "return_text": false}
```

**Response (200):**
```json
{
  "results": [{"index": 0, "score": 0.95, "text": null}, {"index": 1, "score": 0.0001, "text": null}],
  "model": "BAAI/bge-reranker-v2-m3"
}
```

**Error responses:**
- `503` - reranker is disabled

### GET /health

```json
{"status": "ok", "postgres": "connected", "embedding": "healthy", "reranker": "healthy", "memgraph": "connected"}
```

Possible values per service: `"connected"` / `"healthy"` / `"disabled"` / `"error: ..."`.
Overall status: `"ok"` if all non-disabled services are healthy, `"degraded"` otherwise.

### GET /health/models

Returns full model info from TEI for both embedding and reranker (model ID, dtype, max tokens, version, etc.).

## Startup order

1. **Postgres** starts first. `init.sql` runs on first boot only (creates schema). Healthcheck: `pg_isready`. Start period: 30s.
2. **TEI embed** starts in parallel. Downloads model on first boot (1-3 min). Healthcheck: `GET /health`. Start period: 180s. Retries: 60.
3. **TEI rerank** (if `--profile rerank`) starts in parallel. Same download behavior.
4. **Memgraph** (if `--profile graph`) starts in parallel.
5. **API** starts **last** -`depends_on` waits for Postgres and TEI embed to be healthy.
6. On startup, API probes reranker and memgraph:
   - Reachable → connects, logs info
   - Unreachable → logs warning, disables, continues
   - **API never crashes due to missing optional services**

If the reranker finishes downloading after the API starts:
```bash
docker compose restart api
```

## Verified behavior

- Ingest is atomic: document + chunks + vectors in one transaction. If embedding fails, nothing is committed.
- Chunk inserts use batch `executemany`.
- Hybrid search runs semantic + keyword in parallel, merges with RRF.
- RRF correctly boosts results found by multiple retrievers (dual-match scores ~2x single-match).
- Reranker amplifies relevance gap: relevant docs score 0.95+, irrelevant score <0.001.
- Concepts are upserted by name (MERGE) - no duplicates.
- Self-referencing relations are blocked (400).
- Cypher injection is prevented by sanitizing relation types to `[A-Za-z0-9_]` only.
- SQL injection is safe (asyncpg parameterized queries throughout).
- Graph endpoints return 503 when Memgraph is disabled.
- All ports configurable, all bound to 127.0.0.1 (not exposed to network).
- HNSW ef_search is set per-connection via pool init (all connections get the setting).
- Empty/whitespace content produces document with 0 chunks (no crash).
- Unicode, emoji, CJK characters handled correctly.
- Document delete cascades to chunks (no orphan data).
- Deleted documents immediately disappear from search results.
- 404 returned for non-existent resources on GET and DELETE.
- 422 returned for invalid input (empty title, empty content, empty query).
- Empty database returns empty results (no crash).
- Ingest-then-search: no delay needed, results available immediately.
- Cross-domain isolation: legal, medical, tech queries each find their own domain.
- Data persists across `docker compose down`.
- Data destroyed only by `docker compose down - v`.
- Model cache persists on host filesystem across container rebuilds.
- OpenAPI docs auto-generated at `/docs` (21 endpoints).

## Recommendations for best retrieval quality

**Contextual Chunk Headers (built-in)**: rag-base automatically prepends document title and metadata to each chunk before embedding. Improves retrieval by ~28%. No configuration needed. Stored chunk text remains original (headers only affect the embedding vector).

**Contextual Retrieval (client-side)**: For further improvement, implement Anthropic's Contextual Retrieval before ingesting. Use an LLM to generate a 50-100 token chunk-specific summary, prepend it to the chunk content, then send to `POST /v1/documents`. Up to 49% fewer retrieval failures. Requires LLM calls at ingest time, so this is a client concern, not built into rag-base.

## Known limitations

- Graph entity extraction from search queries is naive (word matching). Production use should add NER or LLM-based extraction.
- Graph results are not yet fused into RRF with chunk results (concepts don't map directly to chunks). Graph expansion enriches retrieval context but doesn't contribute to chunk ranking scores.
- Chunking uses word count as token approximation. For precise token counting, add a tokenizer.
- Default chunk size (512 words) means most short documents (<500 words) become a single chunk. Lower `CHUNK_SIZE` for finer-grained retrieval.
- No authentication. Add a reverse proxy (nginx, caddy) for production.
- Single-node only. No clustering or replication.
- TEI CPU image only. For GPU acceleration, swap image tag to CUDA/ROCm variants.
- If reranker profile starts after API, requires `docker compose restart api` to connect.
- Memgraph stores metadata as JSON strings (no native dict type). Serialized/deserialized transparently.
