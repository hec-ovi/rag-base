# rag-base - Technical Reference

## Purpose

Standalone containerized RAG backend. Hybrid retrieval (semantic + keyword + graph) with embedding, reranking, knowledge-graph extraction (LightRAG), and a graph-only fast-mode endpoint. Exposed as a REST API.

## Why this exists

Every RAG project needs the same infrastructure: vector storage, embeddings, BM25, reranking, a knowledge graph, NER. This repo solves all of that once as a plug-and-play service. Future projects (legal, medical, internal docs) spin this up and only write their wrapper / business logic on top.

---

## How it works

### Ingest pipeline

1. Client sends document to `POST /v1/documents`.
2. API chunks the text (paragraph boundaries, configurable `CHUNK_SIZE` / `CHUNK_OVERLAP`). Each chunk also carries a markdown header path breadcrumb (`Section > Subsection`) computed at ingest (Phase 3c).
3. **Contextual Chunk Headers (CCH).** Each chunk's `indexed_content` is built as `[title | metadata] [Header > Path] <chunk text>`. This is the only form fed to the embedder and to the BM25 index. The original raw chunk text is stored in `chunks.content` for display and reconstruction.
4. API embeds all `indexed_content` strings in one batch via TEI (`POST /embed`).
5. API stores document + chunks + vectors in Postgres in a **single atomic transaction**. The BM25 index updates automatically via ParadeDB `pg_search` because the index is defined on `chunks.indexed_content`.
6. If LightRAG is configured (Memgraph + a reachable LLM), the document is also passed to LightRAG for entity-and-relation extraction. LightRAG writes the entity nodes (`:base` label) and edges (`DIRECTED` type) into Memgraph, plus its KV store on disk under `/app/lightrag_data`. This call is awaited inline so retrieval immediately sees graph hits, but failure does NOT block document creation: the rag-base hybrid + rerank path still works.
7. Test-only escape hatch: header `X-LightRAG-Ingest: false` skips the LightRAG step (used by tests that don't assert on graph behavior to avoid paying ~1.5 minutes of LLM time per ingest).

### Retrieval pipeline (`POST /v1/search`)

1. Client sends query.
2. API embeds the query via TEI (`POST /embed`).
3. Up to **three retrievers run in parallel**:
   - **Semantic**: pgvector cosine similarity (`embedding <=> query_vector`) over `chunks.embedding`.
   - **Keyword**: ParadeDB `pg_search` BM25 (`indexed_content @@@ <query>`) - real BM25 with IDF, length normalization, and term-frequency saturation, all evaluated by Tantivy embedded in Postgres.
   - **Graph** (optional, requires Memgraph + LLM): the LLM extracts entity names from the query in concise mode; we Cypher-match those names plus their 1-hop neighbors in Memgraph; we collect the matched entities' `source_id` fields, split on `<SEP>` to recover LightRAG chunk hashes, look those up in LightRAG's `text_chunks` KV to find the original `(full_doc_id, chunk_order_index)` pairs, then run a doc-restricted semantic search over those documents. The resulting chunks are tagged `source="graph"`.
4. Results merged via **Reciprocal Rank Fusion** (RRF) with the standard `1 / (60 + rank)` formula. Results that surface from multiple retrievers get boosted; this is what makes hybrid retrieval beat any single channel.
5. Top `rerank_candidates` (default 50) are cross-encoder-reranked via TEI (`POST /rerank`) - the most accurate but slowest scorer, run last on a bounded candidate pool.
6. Final `top_k` returned.

### Graph-only fast mode (`POST /v1/search/graph`)

A separate endpoint that bypasses pgvector / BM25 / RRF / cross-encoder rerank entirely:

1. **NER**: GLiNER (`knowledgator/gliner-multitask-v1.0`) runs over the query string. No LLM call at query time.
2. **Match**: each NER hit is matched to a `:base` node in Memgraph by `entity_id` case-insensitive. If `fuzzy=true`, falls back to `CONTAINS`.
3. **Subgraph**: 0/1/2-hop traversal over the matched seeds. Two queries (nodes then edges) to dodge a Memgraph quirk where path-comprehension over `r.description` collapsed across edges.
4. **Rank**: `degree` (seeds first, then non-seeds by degree desc) or `none` (preserve match order).
5. **Bridge to chunks**: each node carries `source_id = "chunk-<hash><SEP>chunk-<hash2>..."`. Split on `<SEP>`, look up hashes in LightRAG's `text_chunks` KV, parse `full_doc_id="doc_<int>"` and `chunk_order_index`, then resolve to `chunks.id` via the `(document_id, chunk_index)` unique pair in Postgres.
6. **Return**: structured `{matched_entities, subgraph: {nodes, edges}, chunks, trace}` payload.

This endpoint is the right tool when latency matters (no embedding, no LLM, no rerank), when the query is entity-anchored (a place, a person, a product), or when callers want auditability over the graph traversal.

### Why the order in the hybrid endpoint

- Parallel fan-out in stage 1: each retriever catches what others miss.
- RRF before reranking: the reranker needs to see ALL candidates to make the best decision.
- Rerank last: cross-encoder is the most accurate scoring but slowest, so only run it on the merged top-N, not on everything.

## Containers

### postgres (paradedb/paradedb:0.23.1-pg17)

- **Port**: 5433 host -> 5432 internal (configurable via `POSTGRES_PORT`).
- **Extensions** (loaded by `postgres/init.sql`):
  - `vector` (pgvector): vector type + HNSW index.
  - `pg_search` (ParadeDB / Tantivy): real BM25 inverted index. Replaces the older `tsvector` + `ts_rank` keyword approach.
  - `pg_trgm`: trigram similarity (kept available for fuzzy queries; not on the hot path).
- **Schema** (see `postgres/init.sql`):
  - `documents (id, title, content, metadata jsonb, created_at, updated_at)`
  - `chunks (id, document_id, chunk_index, content, indexed_content, token_count, embedding vector(1024), metadata jsonb, created_at)`
- **Indexes**:
  - HNSW on `chunks.embedding` (cosine distance) with `m=16, ef_construction=64`.
  - `idx_chunks_bm25` on `chunks.indexed_content` via `pg_search` (`USING bm25 (id, indexed_content) WITH (key_field = 'id')`).
  - GIN on `documents.metadata`.
  - Unique on `(chunks.document_id, chunks.chunk_index)`.
- **Persistence**: bind-mounted to `./data/postgres` (host) -> `/var/lib/postgresql/data` (container). `PGDATA=/var/lib/postgresql/data/pgdata` to dodge the initdb-on-existing-dir issue.
- **Init**: `postgres/init.sql` runs on first boot only via `/docker-entrypoint-initdb.d/`.

### embedding (ghcr.io/huggingface/text-embeddings-inference:cpu-1.9)

- **Port**: 8081 host -> 80 internal (configurable via `EMBEDDING_PORT`).
- **Default model**: `Snowflake/snowflake-arctic-embed-l-v2.0` (568M params, 1024 dims). On this development machine the model is set to `BAAI/bge-m3` (also 1024 dims) via `.env`. Both are drop-in compatible from the api perspective.
- **Model cache**: host dir `EMBEDDING_MODELS_DIR` mounted at `/data` inside the container.
- **First boot**: downloads model from HuggingFace Hub (~1.2 GB). Takes 1-3 min. Cached after.
- **TEI endpoints**: `POST /embed`, `POST /v1/embeddings` (OpenAI-compatible), `GET /health`, `GET /info`.

### reranker (ghcr.io/huggingface/text-embeddings-inference:cpu-1.9)

- **Port**: 8082 host -> 80 internal (configurable via `RERANK_PORT`).
- **Default model**: `BAAI/bge-reranker-v2-m3` (568M params, multilingual).
- **Docker profile**: `rerank` - only starts with `docker compose --profile rerank up -d`.
- **Optional**: api detects its absence and skips the rerank step gracefully.
- **TEI endpoints**: `POST /rerank` (request: `{"query": "...", "texts": ["...", "..."]}` -> `[{"index": 0, "score": 0.95}, ...]`), `GET /health`, `GET /info`.

### memgraph (memgraph/memgraph-mage:3.9.0)

- **Port**: 7687 host -> 7687 internal (configurable via `MEMGRAPH_PORT`).
- **Protocol**: Bolt only (no REST). The api container connects via the `neo4j` Python async driver pointing at `bolt://memgraph:7687`.
- **Docker profile**: `graph` - only starts with `docker compose --profile graph up -d`.
- **Optional**: api detects its absence and: (a) graph endpoints return 503, (b) the search graph channel is silently disabled, (c) LightRAG ingest is skipped.
- **Algorithms via MAGE**:
  - `CALL pagerank.get() YIELD node, rank`
  - `CALL community_detection.get() YIELD node, community_id`
  - BFS shortest path: `MATCH p = (a)-[*BFS]-(b) RETURN p`
- **Persistence**: bind-mounted to `./data/memgraph` (host) -> `/var/lib/memgraph` (container).
- **First-boot prereq**: Memgraph runs as UID 101 inside the container. The host bind dir must be owned by 101 or Memgraph will SIGSEGV at startup with banner-only logs:
  ```bash
  docker run --rm -v "$PWD/data/memgraph":/d alpine chown -R 101:101 /d
  ```
- **Schema**:
  - LightRAG-managed entity nodes (`:base` label) with properties `{entity_id, entity_type, description, source_id}` where `source_id = "chunk-<hash><SEP>chunk-<hash2>..."`.
  - LightRAG-managed edges (`DIRECTED` type) with `{description, keywords, weight, source_id}`.
  - The legacy `Concept` / typed-relation API (`/v1/concepts`, `/v1/relations`, `/v1/graph/*`) writes `:Concept` nodes with custom relation labels, kept for tooling and direct graph access alongside the LightRAG-managed graph.
  - `:Concept(name)` unique constraint is created at api startup.

### api (custom, python:3.12-slim-bookworm)

- **Port**: 5050 host -> 8000 internal (configurable via `API_PORT`).
- **Stack**: FastAPI, asyncpg (Postgres), httpx (TEI + LLM Responses), neo4j async driver (Memgraph), uvicorn + uvloop. GLiNER (CPU torch) for query-time NER. LightRAG (`lightrag-hku>=1.4.15`) for ingest-time entity/relation extraction.
- **Depends on**: postgres (healthy) and embedding (healthy) - waits for both before starting.
- **Optional services**: probes reranker, Memgraph, and LLM endpoint at startup. If unreachable, logs a warning and disables them. **Does not crash.**
- **Persistence**: LightRAG KV + nano-vectordb files are bind-mounted to `./data/lightrag` (host) -> `/app/lightrag_data` (container). Without this, `docker compose build api` would silently wipe the chunk-hash -> full_doc_id mapping that bridges Memgraph entity nodes back to Postgres chunks.
- **Healthcheck**: `GET /health` on internal port 8000.

## Startup order

1. **Postgres** starts first. `init.sql` runs on first boot (creates schema, registers extensions). Healthcheck: `pg_isready`. Start period: 30s.
2. **TEI embed** starts in parallel. Downloads model on first boot (1-3 min). Healthcheck: `GET /health`. Start period: 180s. Retries: 60.
3. **TEI rerank** (if `--profile rerank`) starts in parallel. Same download behavior.
4. **Memgraph** (if `--profile graph`) starts in parallel.
5. **API** starts last. Docker `depends_on` waits for Postgres and TEI embed to be healthy.
6. On startup, the api probes reranker, Memgraph, and the LLM endpoint:
   - Reachable -> connects, logs info.
   - Unreachable -> logs warning, disables, continues.
   - **API never crashes due to missing optional services.**
7. The GLiNER model is lazy-loaded on the first `/v1/search/graph` call (not at startup), so the api comes up fast even on a cold cache.

If the reranker finishes downloading after the api starts:
```bash
docker compose restart api
```

## API Reference

Base URL: `http://localhost:5050`
OpenAPI docs: `http://localhost:5050/docs`

### POST /v1/documents

Ingest a document. Chunks it, embeds each chunk's `indexed_content`, stores everything atomically. If LightRAG is configured, also extracts entities/relations into Memgraph + LightRAG KV.

**Request:**
```json
{
  "title": "React Documentation",
  "content": "React is a JavaScript library for building user interfaces...",
  "metadata": {"source": "docs", "language": "en"},
  "contextual_retrieval": false
}
```

| Field | Type | Default | Description |
|---|---|---|---|
| `title` | string | *(required)* | Document title; min length 1 |
| `content` | string | *(required)* | Document body; min length 1 |
| `metadata` | object | `{}` | Free-form JSONB metadata (becomes searchable via document filters) |
| `contextual_retrieval` | bool | `false` | When true, runs Anthropic Contextual Retrieval per chunk: the LLM at `LLM_BASE_URL` generates a 50-100 token "situating" blurb per chunk, prepended to `chunks.indexed_content` (raw `chunks.content` stays untouched). Requires the LLM endpoint to be reachable; on per-chunk LLM failure that chunk gets no blurb but ingest still succeeds. Prompt body lives in `api/prompts/contextual_retrieval.md` (user-editable). |

**Response (201):**
```json
{
  "id": 1,
  "title": "React Documentation",
  "content": "React is a JavaScript library for building user interfaces...",
  "metadata": {"source": "docs", "language": "en"},
  "created_at": "2026-04-27T10:29:22.526748Z",
  "updated_at": "2026-04-27T10:29:22.526748Z",
  "chunk_count": 1
}
```

**Headers:**
- `X-LightRAG-Ingest: false` skips LightRAG entity extraction (test-only escape hatch).

**Error responses:**
- `422` - empty title or empty content.

### GET /v1/documents?offset=0&limit=20

List documents, paginated. Returns array of document objects with `chunk_count`.

### GET /v1/documents/{id}

Get a document with all its chunks.

**Response (200):**
```json
{
  "id": 1,
  "title": "React Documentation",
  "content": "...",
  "metadata": {...},
  "created_at": "...",
  "updated_at": "...",
  "chunk_count": 1,
  "chunks": [
    {"id": 1, "chunk_index": 0, "content": "React is a JavaScript...", "token_count": 85}
  ]
}
```

**Error responses:**
- `404` - document not found.

### DELETE /v1/documents/{id}

Delete document and all its chunks (CASCADE). Returns `204 No Content`. Does NOT delete LightRAG-managed graph nodes (those persist by design and may be referenced by other documents).

**Error responses:**
- `404` - document not found.

### POST /v1/search

Hybrid search: semantic + keyword + graph -> RRF -> optional rerank.

**Request:**
```json
{
  "query": "frontend frameworks for building UI",
  "top_k": 10,
  "rerank": true,
  "rerank_candidates": 50,
  "min_score": 0.0,
  "include_graph": true
}
```

| Field | Type | Default | Description |
|---|---|---|---|
| `query` | string | *(required)* | Search query text |
| `top_k` | int | 20 | Number of results to return (1-100) |
| `rerank` | bool | true | Use cross-encoder reranking (falls back if reranker disabled) |
| `rerank_candidates` | int | 50 | How many RRF-merged results to send to reranker |
| `min_score` | float | 0.0 | Minimum cosine similarity for semantic results |
| `include_graph` | bool | true | Include the LightRAG graph channel (ignored if Memgraph or LLM disabled) |

**Response (200):**
```json
{
  "query": "frontend frameworks for building UI",
  "results": [
    {
      "chunk_id": 4,
      "document_id": 5,
      "document_title": "React Documentation",
      "content": "React is a JavaScript library for building user interfaces...",
      "score": 0.9924,
      "sources": ["semantic", "keyword"]
    }
  ],
  "total": 3,
  "retrievers_used": ["semantic", "keyword", "graph", "rerank"]
}
```

`sources` shows which retrievers contributed to each result. `retrievers_used` shows all retrievers that ran.

### POST /v1/search/semantic

Vector-only search. Same request/response shape. No keyword, graph, or reranking.

### POST /v1/search/keyword

Keyword-only search (BM25 over `indexed_content`). Same request/response shape. No vector, graph, or reranking.

### POST /v1/search/graph

Graph-only fast-mode retrieval. NER (GLiNER) -> Memgraph entity match -> 0/1/2-hop traversal -> structured response. Bypasses pgvector, BM25, RRF, and the cross-encoder reranker entirely. No LLM at query time.

**Request:**
```json
{
  "query": "what does Acme Robotics work on?",
  "max_entities": 8,
  "hops": 1,
  "ranking": "degree",
  "top_k_chunks": 20,
  "fuzzy": true,
  "ner_labels": null
}
```

| Field | Type | Default | Description |
|---|---|---|---|
| `query` | string | *(required)* | Search query text |
| `max_entities` | int | 8 | Cap on NER hits to consider as seeds |
| `hops` | int | 1 | Subgraph traversal depth (0, 1, 2) |
| `ranking` | enum | `"degree"` | Node ranking strategy: `"degree"` or `"none"` |
| `top_k_chunks` | int | 20 | Max chunks to bridge from the subgraph |
| `fuzzy` | bool | true | Fall back to CONTAINS match when exact_ci finds nothing |
| `ner_labels` | list[str]\|null | null | Override default GLiNER labels (`person`, `organization`, `location`, ...) |

**Response (200):**
```json
{
  "query": "what does Acme Robotics work on?",
  "matched_entities": [
    {"id": "Acme Robotics", "name": "Acme Robotics", "type": "ORGANIZATION", "match_score": 0.94, "match_method": "exact_ci", "source_phrase": "Acme Robotics"}
  ],
  "subgraph": {
    "nodes": [{"id": "Acme Robotics", "name": "Acme Robotics", "type": "ORGANIZATION", "description": "...", "degree": 3}],
    "edges": [{"source": "Acme Robotics", "target": "Alice Chen", "relation": "EMPLOYS", "weight": 1.0, "keywords": "..."}]
  },
  "chunks": [
    {"chunk_id": 17, "doc_id": 5, "lightrag_chunk_hash": "chunk-abc123...", "text": "...", "source_entities": ["Acme Robotics"]}
  ],
  "trace": {
    "latency_ms": 412.5, "ner_ms": 88.4, "graph_ms": 51.2, "chunk_bridge_ms": 272.9,
    "n_ner_entities": 1, "n_matched_nodes": 1, "n_subgraph_nodes": 4, "n_subgraph_edges": 3
  }
}
```

200 with empty `matched_entities` and `chunks` when NER finds nothing or no graph nodes match. 503 only when a hard infra dep is unreachable (Memgraph driver missing, NER service not initialized).

### POST /v1/concepts

Create or update a concept node in the knowledge graph (`:Concept` label, distinct from LightRAG's `:base` entities). Upserts by name - if a concept with the same name exists, it updates type/description/metadata instead of creating a duplicate.

**Request:**
```json
{"name": "React", "type": "Framework", "description": "JavaScript UI library", "metadata": {}}
```

**Response (201):**
```json
{"id": 9, "name": "React", "type": "Framework", "description": "JavaScript UI library", "metadata": {}}
```

### GET /v1/concepts/{id}

Get a concept with all its outgoing relations.

**Response (200):**
```json
{
  "id": 9,
  "name": "React",
  "type": "Framework",
  "description": "JavaScript UI library",
  "metadata": {},
  "relations": [
    {"relation_type": "DEPENDS_ON", "target": "JavaScript", "target_id": 10}
  ]
}
```

**Error responses:**
- `404` - concept not found.
- `503` - Memgraph is disabled.

### DELETE /v1/concepts/{id}

Delete concept and all its edges (DETACH DELETE). Returns `204`.

### POST /v1/relations

Create a typed directed edge between two `:Concept` nodes (by name).

**Request:**
```json
{"source_name": "React", "target_name": "JavaScript", "relation_type": "DEPENDS_ON", "metadata": {}}
```

**Error responses:**
- `400` - self-referencing relation (source_name == target_name).
- `404` - one or both concepts not found.
- `503` - Memgraph is disabled.

### GET /v1/relations?concept_name=React

Get all relations involving a concept (both directions).

### DELETE /v1/relations/{id}

Delete a relation. Returns `204`.

### GET /v1/graph/neighbors/{concept_id}?depth=2

Multi-hop neighborhood traversal over the `:Concept` graph. Returns all concepts within N hops (max 5).

**Response (200):**
```json
[
  {"id": 10, "name": "JavaScript", "type": "Language", "depth": 1},
  {"id": 12, "name": "Node.js", "type": "Runtime", "depth": 2}
]
```

### GET /v1/graph/path/{from_id}/{to_id}

Shortest path between two concepts (BFS).

**Response (200):**
```json
[
  {"id": 11, "name": "Next.js", "type": "Framework"},
  {"id": 9, "name": "React", "type": "Framework"},
  {"id": 10, "name": "JavaScript", "type": "Language"}
]
```

**Error responses:**
- `404` - no path exists between the two concepts.

### GET /v1/graph/communities

Louvain community detection over the `:Concept` graph. Returns every concept with its community ID.

**Response (200):**
```json
[
  {"id": 13, "name": "PostgreSQL", "type": "Database", "community_id": -1},
  {"id": 9, "name": "React", "type": "Framework", "community_id": 0},
  {"id": 10, "name": "JavaScript", "type": "Language", "community_id": 0}
]
```

Isolated nodes get `community_id: -1`.

### GET /v1/graph/stats

```json
{"concepts": 6, "relations": 6}
```

Rerank candidates directly via TEI. Passthrough.

**Request:**
```json

```

**Response (200):**
```json

```

**Error responses:**
- `503` - reranker is disabled.

### GET /health

```json
{"status": "ok", "postgres": "connected", "embedding": "healthy", "reranker": "healthy", "memgraph": "connected"}
```

Possible values per service: `"connected"` / `"healthy"` / `"disabled"` / `"error: ..."`.
Overall status: `"ok"` if all non-disabled services are healthy, `"degraded"` otherwise.

### GET /health/models

Returns full model info from TEI for both embedding and reranker (model id, dtype, max tokens, version, etc.).

## Prompt templates (`api/prompts/`)

LLM prompt bodies are decoupled from Python code and live in `api/prompts/*.md`. Two prompts ship today:

| File | Used by | Variables |
|---|---|---|
| `api/prompts/contextual_retrieval.md` | `services/contextual_retrieval.py` when `POST /v1/documents` runs with `contextual_retrieval: true` | `{{document}}`, `{{chunk}}` |
| `api/prompts/query_entity_extraction.md` | `services/lightrag_store.py:extract_query_entities` (graph channel of `POST /v1/search`) | `{{query}}` |

Each file may begin with an HTML comment block (`<!-- ... -->`) used as inline editor docs; the loader strips that header before the template reaches the LLM. Substitution is a single regex pass: `{{name}}` placeholders are replaced once, substituted values are NOT re-scanned (so a document that contains literal `{{chunk}}` is harmless).

Templates are loaded once at first access and cached for the process lifetime. To pick up an edit:

```
docker compose restart api
```

Adding a new prompt: drop a new `.md` under `api/prompts/`, render with `prompts.render("name", **vars)` from `src.services.prompts`. No Python registration needed; missing template names raise `FileNotFoundError` at first use.

## Verified behavior

- Ingest is atomic: document + chunks + vectors in one Postgres transaction. If embedding fails, nothing is committed.
- LightRAG ingest is best-effort: if it fails or hangs past its internal timeout, the doc is still committed; only graph extraction is skipped.
- Contextual Retrieval is opt-in via `contextual_retrieval: true` on `POST /v1/documents`; when enabled, per-chunk LLM failure produces an empty blurb for that chunk only and ingest still succeeds. With the flag false (default) ingest is byte-identical to pre-CR behavior.
- Chunk inserts use batch `executemany`.
- Hybrid search runs semantic + keyword (+ graph if enabled) in parallel, merges with RRF.
- RRF correctly boosts results found by multiple retrievers (dual-match scores ~2x single-match).
- Reranker amplifies relevance gap: relevant docs score 0.95+, irrelevant score <0.001.
- LightRAG entity nodes are merged across documents by `entity_id` (case-sensitive); re-ingesting the same content as a new doc augments `source_id` rather than duplicating nodes.
- The `/v1/search/graph` endpoint is read-only against Memgraph, LightRAG KV, and Postgres. It never writes, never mutates LightRAG state.
- `:Concept` upserts by name (MERGE) - no duplicates.
- Self-referencing relations are blocked (400).
- Cypher injection is prevented by sanitizing relation types to `[A-Za-z0-9_]` only.
- SQL injection is safe (asyncpg parameterized queries throughout).
- Graph endpoints return 503 when Memgraph is disabled.
- All ports configurable, all bound to 127.0.0.1 (not exposed to network).
- HNSW `ef_search` is set per-connection via pool init (all connections get the setting).
- Empty/whitespace content produces a document with 0 chunks (no crash).
- Unicode, emoji, CJK characters handled correctly.
- Document delete cascades to chunks (no orphan data).
- Deleted documents immediately disappear from search results.
- 404 returned for non-existent resources on GET and DELETE.
- 422 returned for invalid input (empty title, empty content, empty query).
- Empty database returns empty results (no crash).
- Ingest-then-search: no delay needed for hybrid + rerank channels; the graph channel becomes available as soon as LightRAG ingest completes for that doc.
- Cross-domain isolation: legal, medical, tech queries each find their own domain.
- Data persists across `docker compose down`, `docker compose down + up`, `docker compose build api`, and even `docker volume prune` - because all persistent state is on host bind mounts under `./data/`, not in named docker volumes.
- Data destroyed only by `rm -rf ./data/{postgres,memgraph,lightrag}`.
- Model cache persists on host filesystem across container rebuilds (separate dirs, separate from `./data/`).
- OpenAPI docs auto-generated at `/docs`.

## Recommendations for best retrieval quality

**Contextual Chunk Headers (built-in)**: rag-base automatically prepends the document title, metadata, and the markdown heading breadcrumb to each chunk's `indexed_content` before embedding and BM25 indexing. This is what the embedder and BM25 actually see; the raw chunk text in `chunks.content` is preserved for display. No configuration needed.

**Contextual Retrieval (built-in, opt-in)**: Send `contextual_retrieval: true` on `POST /v1/documents` to enable Anthropic Contextual Retrieval at ingest. The api makes one LLM call per chunk against `LLM_BASE_URL`, with the full document as a stable prefix so vLLM's automatic prefix caching shares the doc KV across calls (~$1 per 1M document tokens, per Anthropic's measured cost). The 50-100 token blurb gets prepended to `chunks.indexed_content` alongside the title + header path; raw `chunks.content` stays unchanged. Anthropic reports up to 49% fewer retrieval failures with CR alone, 67% with rerank. Lift is corpus-dependent; measure before enabling corpus-wide.

The CR prompt body lives in `api/prompts/contextual_retrieval.md`; edit and `docker compose restart api` to reload (no code change required).

**Use the right endpoint for the query**:
- `/v1/search` for general queries where you want recall + precision.
- `/v1/search/semantic` when paraphrase is the dominant matching mode.
- `/v1/search/keyword` for queries with rare jargon or exact terms (e.g. statute citations, error codes).
- `/v1/search/graph` for entity-anchored queries (a place, a person, a product) where latency matters - no LLM at query time.

## Known limitations

- Single-node only. No clustering or replication.
- TEI CPU image only. For GPU acceleration, swap image tag to CUDA/ROCm variants.
- No authentication. Add a reverse proxy (nginx, caddy) for production.
- LightRAG ingest on the local Qwen3.6-27B-AWQ4 reasoning vLLM takes ~1-3 min per chunk with `enable_thinking=false`. Sustainable for ad-hoc ingest; batch ingest pipelines should run LightRAG asynchronously via the `X-LightRAG-Ingest: false` header at ingest time and then bulk-extract later.
- Memgraph stores metadata as JSON strings (no native dict type). Serialized/deserialized transparently by the api.
- If reranker profile starts after API, requires `docker compose restart api` to connect.
- Default chunk size (512 words) means most short documents (<500 words) become a single chunk. Lower `CHUNK_SIZE` for finer-grained retrieval.
